# Dory 自托管部署

自托管 Dory 适合希望由自己控制部署环境、访问边界、数据库凭据、AI Provider 和应用数据持久化的团队。当前公开仓库主要提供 Docker 优先的部署路径。

Dory 文档里有两类“数据库”需要区分：

| 数据库角色 | 存储内容 | 示例 |
| --- | --- | --- |
| Dory 应用数据库 | Dory 用户、组织、连接、Saved Queries、设置、MCP tokens 和工作区状态。 | PGlite 文件存储或 Postgres。 |
| 被连接的分析数据库 | 团队通过 Dory 查询的外部数据库。 | ClickHouse、PostgreSQL、Neon、MySQL、MariaDB、SQLite、DuckDB、MotherDuck。 |

本页重点说明第一类：Dory 自己的应用数据应该如何持久化。

## 自托管适合谁

- 需要在公司内网或私有云运行 Dory。
- 需要自己管理数据库连接和 AI Key。
- 需要为团队提供统一的浏览器访问入口。
- 需要连接内网数据库、云数据库或测试环境数据库。
- 需要明确数据访问、账号权限和 Secret 管理边界。

## 推荐架构

基础自托管架构通常包括：

1. Dory Docker 容器。
2. 持久化应用数据存储。
3. 反向代理和 HTTPS。
4. Secret 管理系统或受控 `.env`。
5. 可访问目标数据库的网络环境。
6. 邮件服务和 OAuth 配置（如需要）。

## 持久化方案选择

生产上线前，需要先选择 Dory 应用数据库模式。

| 模式 | 适合场景 | 必要配置 |
| --- | --- | --- |
| PGlite 文件存储 | 本地评估、单机自托管、小团队、简单备份。 | `DB_TYPE=pglite`，并把 Docker 的 `/app/data` 挂载到持久化磁盘。 |
| Postgres | 生产团队、已有数据库运维体系、集中备份、监控和恢复。 | `DB_TYPE=postgres`、`DATABASE_URL`，以及可选的 Postgres 连接池 / SSL 配置。 |

不要在没有持久化存储的情况下长期运行 Dory 容器。如果应用数据只写在容器内部文件系统，容器删除或重建后，Dory 工作区数据可能丢失。

## 方案一：PGlite + Docker 持久化磁盘

PGlite 是 Dory 默认的应用存储模式。在 Docker 中，默认 PGlite 路径是 `/app/data/dory`，因此需要把 `/app/data` 挂载到持久化磁盘。

使用 Docker named volume：

```bash
docker volume create dory-data

docker run -d --name dory \
  -p 3000:3000 \
  --env-file .env \
  -e DB_TYPE=pglite \
  -v dory-data:/app/data \
  dorylab/dory:latest
```

如果希望文件明确落在服务器目录中，可以使用 bind mount：

```bash
mkdir -p /srv/dory/data

docker run -d --name dory \
  -p 3000:3000 \
  --env-file .env \
  -e DB_TYPE=pglite \
  -v /srv/dory/data:/app/data \
  dorylab/dory:latest
```

也可以通过 `PGLITE_DB_PATH` 指定 PGlite 文件路径，但该路径应位于已挂载目录内：

```bash
DB_TYPE=pglite
PGLITE_DB_PATH=/app/data/dory
```

升级前请备份挂载目录或 Docker volume。

## 方案二：Postgres 作为生产应用数据库

生产环境可以用 Postgres 作为 Dory 的应用数据库。适合已有托管 Postgres、统一备份策略、监控和恢复流程的团队。

示例 `.env`：

```bash
DB_TYPE=postgres
DATABASE_URL=postgres://dory:strong_password@postgres.example.com:5432/dory
POSTGRES_SSL=true
POSTGRES_SSL_REJECT_UNAUTHORIZED=true
POSTGRES_MAX_CONNECTIONS=10
```

然后使用同一组应用 Secret 和公开访问地址启动 Dory：

```bash
docker run -d --name dory \
  -p 3000:3000 \
  --env-file .env \
  dorylab/dory:latest
```

建议为 Dory 创建独立 Postgres database 和独立数据库用户。该用户需要能够连接数据库、创建和更新 Dory 应用表，并执行 Dory schema migrations。

## PGlite 和 Postgres 怎么选

| 需求 | 选择 PGlite | 选择 Postgres |
| --- | --- | --- |
| 最快完成自托管启动 | 是 | 可以，但配置更多。 |
| 单节点部署 | 是 | 是。 |
| 托管备份和恢复 | 需要手动备份 volume。 | 使用 Postgres 备份体系。 |
| 已有生产数据库运维体系 | 不太适合。 | 是。 |
| 多个应用实例 | 不建议。 | 优先选择 Postgres。 |
| 文件级备份简单 | 是。 | 使用数据库 dump 或服务商快照。 |

## 部署前检查

| 项目 | 检查内容 |
| --- | --- |
| 域名 | 准备 `https://dory.example.com` 之类的访问地址。 |
| HTTPS | 生产环境必须使用 HTTPS。 |
| Secret | 生成 `DS_SECRET_KEY` 和 `BETTER_AUTH_SECRET`。 |
| Auth URL | `BETTER_AUTH_URL` 与公开访问地址一致。 |
| 应用数据库 | 选择 `DB_TYPE=pglite` 并持久化挂载 `/app/data`，或选择 `DB_TYPE=postgres` 并配置 `DATABASE_URL`。 |
| AI | 配置 provider、model、API key 和 URL。 |
| 数据库网络 | Dory 运行环境能访问目标数据库。 |
| 初始化用户 | 配置 `DORY_INIT_USER_EMAIL` 和 `DORY_INIT_USER_PASSWORD`。 |
| 邮件 | 如果启用邮箱验证或邀请，配置 Resend。 |

## 基础部署步骤

1. 准备 `.env`。
2. 选择 PGlite 或 Postgres 作为 Dory 应用存储。
3. 配置持久化：
   - PGlite：把 `/app/data` 挂载到 Docker volume 或宿主机磁盘。
   - Postgres：创建 Dory database，并设置 `DATABASE_URL`。
4. 启动 Docker 容器。
5. 配置 HTTPS 反向代理。
6. 使用初始化用户登录。
7. 修改初始密码或配置团队登录方式。
8. 添加测试数据库连接。
9. 验证 SQL Console、Explorer、AI Chat 和 Saved Queries。
10. 再添加生产数据库连接。

## 生产配置建议

```bash
DS_SECRET_KEY=replace_with_base64_secret
BETTER_AUTH_SECRET=replace_with_hex_secret
BETTER_AUTH_URL=https://dory.example.com
TRUSTED_ORIGINS=https://dory.example.com
NEXT_PUBLIC_REQUIRE_EMAIL_VERIFICATION=true
DORY_INIT_USER_EMAIL=admin@example.com
DORY_INIT_USER_PASSWORD=change_this_password
DB_TYPE=pglite
PGLITE_DB_PATH=/app/data/dory
DORY_AI_PROVIDER=openai
DORY_AI_MODEL=gpt-4o-mini
DORY_AI_API_KEY=replace_with_api_key
DORY_AI_URL=https://api.openai.com/v1
RESEND_API_KEY=replace_with_resend_key
EMAIL_FROM="Dory <noreply@example.com>"
```

## 数据库连接安全

- 生产数据库优先使用只读账号。
- 为 Dory 创建独立数据库用户，避免共用管理员账号。
- 云数据库需要放行 Dory 服务器出口 IP。
- 内网数据库可使用 SSH Tunnel。
- SQLite 和 DuckDB 文件需要通过 volume 挂载到容器。

## 备份和恢复

PGlite 模式：

- 备份挂载到 `/app/data` 的 Docker volume 或宿主机目录。
- 条件允许时，先停止容器再做文件级备份。
- 恢复时，把恢复后的目录重新挂载到 `/app/data`，再启动相同或兼容的新版本 Dory 镜像。

Postgres 模式：

- 使用托管 Postgres 快照、时间点恢复或 `pg_dump`。
- Dory 升级前先备份。
- 在测试环境恢复并验证后，再把备份策略作为生产恢复方案。

## AI 和数据边界

Dory 的 AI 功能会把必要上下文发送给配置的 AI Provider。团队上线前应明确：

- 使用哪个 AI Provider。
- Schema、SQL 和结果摘要是否允许发送给该 Provider。
- AI Key 如何轮换和审计。

## 升级和回滚

升级前：

- 备份持久化数据。
- 备份 `.env`。
- 记录当前镜像版本。
- 在测试环境验证新版本。

升级后：

- 验证登录。
- 验证已有连接。
- 验证 SQL Console 和 AI。
- 验证 Saved Queries 和团队访问。
- 验证 MCP token 和 Agent 访问。

## 不建议的做法

- 使用默认初始化密码长期运行。
- 在公网暴露 HTTP 明文服务。
- 用管理员数据库账号做日常分析。
- 把 AI Key、数据库密码写进镜像或 Git。
- 未确认 `BETTER_AUTH_URL` 就配置 OAuth 或邮件邀请。
