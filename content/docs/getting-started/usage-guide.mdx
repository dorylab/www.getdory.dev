# Dory Usage Guide

Dory is a SQL and AI workbench for databases and local data files. The usual path is:

1. Open Dory.
2. Create or select a data connection.
3. Inspect tables and columns.
4. Run SQL.
5. Review the result table.
6. Switch to Charts when the result is shaped for visualization.
7. Use AI or Saved Queries when the work should continue or repeat.

This page follows that flow using the current product behavior.

## 1. Open Dory

You can use Dory from the desktop app, the hosted product, or a self-hosted deployment.

For local files such as SQLite, DuckDB, CSV, Excel, or Parquet, use the desktop app or a deployment that can access the file path. The path must be readable by the Dory runtime. Dory automatically prepares the default working context for the current user, so there is no separate setup step before connecting data.

## 2. Connect Data

Open **Connections**. You can either create a database connection or open a local file dataset.

### Database Connections

Click **Add Connection**, choose the database type, fill in the required settings, and create the connection.

| Type | Use it for |
| --- | --- |
| ClickHouse | Logs, events, observability, analytics, and monitoring workflows |
| PostgreSQL / Neon | Application data, product analytics, and general SQL work |
| MySQL / MariaDB | Business systems, web applications, and reporting |
| SQLite | Local `.sqlite` / `.db` files, demos, prototypes, and reproducible examples |
| DuckDB / MotherDuck | Local analytics files, embedded analytics, and MotherDuck projects |

For a local SQLite file in Dory desktop, choose **SQLite** and enter an absolute path such as:

```text
/Users/Shared/Data/demo.sqlite
```

![Dory desktop SQLite connection](/images/core-features/dory-desktop-connections-sqlite.png)

### Open Files

Use **Open Files** when the source is a file you want Dory to expose as queryable tables, such as Excel, CSV, or Parquet.

The flow is:

1. Click **Open Files**.
2. Enter a dataset name.
3. Enter the server-accessible **File Path**.
4. Click **Preview**.
5. Review the detected sheets or file relations.
6. Keep the selected sheets/tables, or expand **Advanced options** to customize table names.
7. Create the dataset.

After creation, the dataset appears in Connections and can be used from Explorer and SQL Console.

## 3. Inspect the Data

Before writing SQL, confirm the structure.

Use **Explorer** when you want a full object view: databases, schemas, tables, views, columns, DDL, preview data, and table details.

![Dory Explorer table list](/images/getting-started/dory-usage-explorer.png)

Use the SQL Console left schema panel when you are already writing a query. It shows the current database, table list, and quick actions such as inserting a starter `SELECT`.

![Dory Explorer table preview](/images/getting-started/dory-usage-explorer-table-preview.png)

Check these before analysis:

- The expected database or schema is selected.
- The target tables are visible.
- Column names and types match the query you plan to write.
- Preview rows look like the data you expect.

## 4. Run SQL

Open **SQL Console** with the correct connection selected.

A reliable first query is a small preview:

```sql
select *
from orders
limit 100;
```

For analysis, shape the result before charting. For example:

```sql
select strftime('%Y-%m', created_at) as month,
       status,
       round(sum(amount), 2) as revenue,
       count(*) as orders
from orders
group by month, status
order by month, status;
```

After running, check:

- Whether the query finished successfully.
- Execution time.
- Returned row count.
- Column names and values.
- Whether the output matches the analysis question.

![Dory desktop SQL Console SQLite result](/images/core-features/dory-desktop-sql-console-sqlite-results.png)

## 5. Review Results and Build Charts

Results open in the result area under the SQL editor.

Use **Table** first to validate raw values. Use **Charts** after the result is already shaped for visualization.

Good chart inputs usually look like this:

| Goal | SQL output shape |
| --- | --- |
| Trend | Time column + numeric metric |
| Comparison | Category column + numeric metric |
| Grouped trend | Time column + group column + numeric metric |
| Detail review | IDs, timestamps, statuses, and useful diagnostic columns |

If the chart is not useful, change the SQL first. Usually the fix is to aggregate earlier, rename columns clearly, remove extra fields, or sort by time/category.

![Dory desktop SQL Console SQLite chart](/images/core-features/dory-desktop-sql-console-sqlite-charts.png)

## 6. Use AI When It Helps

Dory has two AI entry points. Use **Chatbot** from the left sidebar for open-ended questions about the selected connection. Use **Ask AI** in SQL Console when you want help with the current query, result, or chart-ready follow-up.

Use AI for:

- Drafting SQL from a natural-language request.
- Explaining an unfamiliar query.
- Fixing SQL errors.
- Rewriting SQL for a chart-friendly result.
- Continuing analysis from the current result.

![Dory AI Chat question and answer](/images/getting-started/dory-usage-ai-chat-question-answer.png)

Review AI-generated SQL before running it, especially on production databases. When the goal is a chart, ask AI for a result shaped with clear time/category and metric columns.

## 7. Save Reusable Work

When a query is validated and likely to be reused, click **Save Query**.

Save queries that are:

- Recurring metrics.
- Debugging or operations checks.
- Shared team analysis.
- Inputs for dashboards, charts, or repeated reports.

![Dory Save Query dialog](/images/getting-started/dory-usage-save-query.png)

Use names that describe the business question, not just the table name. For example, prefer `Monthly revenue by status` over `orders query`.

## 8. Use ClickHouse-Specific Workflows

ClickHouse connections can expose additional operations when the connected user has permission:

- **Monitoring** for recent queries, slow queries, error queries, latency, QPM, and query detail.
- **Privileges** for users, roles, global grants, database/table/view privileges, and cluster-aware operations.

Use least-privilege credentials for production. Monitoring and privilege workflows depend on ClickHouse system tables and account permissions.

## 9. Connect External AI Agents

Dory supports MCP / Agent Access for controlled access from tools such as Codex or Claude.

Common MCP use cases:

- List available Dory connections.
- Read schema and table metadata.
- Preview tables.
- Run restricted read-only SQL.
- Use database context in an external agent workflow.

Start from [MCP / Agent Access](/en/docs/reference/mcp) when you need this setup.
